# Audit findings — 2026-05-27

Combined security audit and code review of SkyNetControl. Findings are classified by severity and tracked through to fix. Each finding cites file:line and includes the remediation taken.

## Critical

### H1 — `GET /api/config/` exposes secrets to any authenticated user

**File:** `backend/config_mgmt/routes.py:18-23`

`GET /api/config/` requires `get_current_user`, which only rejects `DELETED` users. The endpoint returns the raw `AppConfig` table dump including `claude_api_key`, `delivery.groupsio.api_key`, `callbook.*.password`, and any other secret stored there. Any PENDING user (just completed OAuth, not yet approved) or VIEWER can exfiltrate every secret.

**Fix:** Gate behind `require_role(UserRole.ADMIN)` to match the PUT endpoint. **Status: Fixed.**

### H2 — `anonymize_user` fails on the second call due to a unique constraint

**File:** `backend/privacy/service.py:96`, `backend/auth/models.py`

`anonymize_user` sets the anonymized user's `oidc_subject` to the literal string `"deleted"`. The `users.oidc_subject` column has `unique=True`. The second anonymization throws `IntegrityError` mid-transaction, leaving the database in a partially-mutated state (PATs deleted, audit log written, but user record not anonymized).

**Fix:** Use a unique placeholder per user, e.g., `f"deleted:{anon_id}"`. Add a regression test that anonymizes two distinct users. **Status: Fixed.**

## High

### M1 — Secrets persist in the audit log

**File:** `backend/config_mgmt/routes.py:34`

The config-update route writes `{"key": key, "value": body.value}` into `audit_log.details`. When an admin sets `claude_api_key` or any other sensitive value, the full secret is preserved indefinitely in the audit log. The audit log is admin-readable only, but this means rotating a key doesn't remove old copies.

**Fix:** Maintain a `SENSITIVE_KEYS` set in the routes module; for matching keys, log `"[REDACTED]"` instead of the value. **Status: Fixed.**

### M3 — PENDING users bypass the anonymous-only filter on public endpoints

**Files:** `backend/modules/checkins/routes.py:130-135`, `backend/modules/schedule/routes.py:212-231`

`GET /api/checkins/session/{id}` and `GET /api/schedule/sessions` both have an `if user is None` guard to restrict anonymous viewers to completed sessions. The optional-auth helper returns the User object for PENDING users (their OAuth login succeeded), so the guard skips and PENDING users see all session statuses — including scheduled and in-progress sessions they shouldn't see until approved.

**Fix:** Change both guards to `if user is None or user.role == UserRole.PENDING`. PENDING is treated like anonymous for the purposes of these public read endpoints. **Status: Fixed.**

### M5 — First-user-becomes-admin has a race window on PostgreSQL

**File:** `backend/auth/routes.py:120-121`

`user_count = db.query(User).count()` followed by an insert is not atomic. On PostgreSQL with concurrent OAuth callbacks (two users finishing login at the same time), both can see `count == 0` and both insert as ADMIN. The `oidc_subject` unique constraint prevents two rows with the same subject, but two distinct subjects can both become ADMIN. SQLite serializes writes so the race window doesn't open there.

**Fix:** Use a `SELECT … FOR UPDATE` advisory lock pattern, or query/insert in a single atomic statement. Since the deployment default is SQLite and the production target is single-tenant, the simplest fix is documenting the constraint plus using a row-level lock on PostgreSQL. **Status: Fixed (added `with_for_update()` guard around the count, no-op on SQLite).**

### M6 — Default JWT secret allows token forgery if not overridden

**File:** `backend/config.py:30`

`jwt_secret_key: str = "change-me-in-production"`. An operator who forgets to set `SKYNET_JWT_SECRET_KEY` runs with a known-public string, so anyone who can read the source can forge valid JWTs for any callsign. There's no startup check.

**Fix:** In `create_app`, fail fast with a clear error if the JWT secret is still the default value. **Status: Fixed.**

### Frontend — `apiFetch` missing `credentials: "include"`

**File:** `frontend/src/api/client.ts:18-24`

`fetch` is called without explicit credentials. In the current deployment topology (production: single-origin via Uvicorn; dev: same-origin via Vite proxy) cookies are sent automatically. The omission is not a current bug, but it's defense-in-depth — any future change that introduces a cross-origin path (subdomain, separate hosting) would silently break auth. Adding `credentials: "include"` is a one-line, no-op-for-now defensive change.

**Fix:** Add `credentials: "include"` to the fetch options. **Status: Fixed.**

## Medium

### M2 — No rate limiting on auth, Claude chat, or mailbox scan endpoints

**Files:** `backend/auth/routes.py`, `backend/modules/activities/routes.py`, `backend/modules/checkins/routes.py`

No throttling on `/api/auth/login/{provider}`, the Claude chat endpoint (each call hits the Anthropic API), or `/api/checkins/scan/{session_id}` (reads the mailbox directory). A compromised net_control account could drain Claude API credits; a malicious or buggy scanner trigger could cause heavy filesystem IO.

**Fix:** Deferred. Adding a rate limiter pulls in a new dependency (`slowapi` or similar) and a Redis/in-memory store. For a single-tenant small-net deployment this is acceptably low-risk. Tracked here for the future. **Status: Deferred (low priority for current deployment scale).**

### L1 — Manual scan and background scanner use different mailbox paths

**Files:** `backend/modules/checkins/routes.py:112`, `backend/integrations/scanner/service.py:69`

The manual `/api/checkins/scan/{id}` route reads `mailbox_path` directly. The background scanner reads `{mailbox_path}/in`. For the same config value, the two paths process different file sets — leading to "manual scan finds messages, auto-scan doesn't" or vice versa.

**Fix:** Make both consistent. The background scanner's `/in` convention matches PAT's actual layout, so update the manual route to also append `/in`. **Status: Fixed.**

### L3 — OAuth state cookie missing `secure` flag on HTTPS deployments

**File:** `backend/auth/routes.py:57-63`

The `oauth_state` cookie is set with `httponly=True, samesite="lax"` but no `secure`. On HTTPS deployments this means it could be sent over a plaintext HTTP request (e.g., if an attacker downgrades the connection or a misconfigured proxy serves HTTP). The `access_token` cookie correctly applies `secure=True` when `app_base_url.startswith("https://")`.

**Fix:** Mirror the same `is_secure` conditional on the state cookie. **Status: Fixed.**

### L8 — Claude API error detail forwarded to the client

**File:** `backend/modules/activities/routes.py:240`

`raise HTTPException(status_code=502, detail=f"Claude API error: {exc}")` exposes raw Anthropic SDK exception messages to the client. Even for admin/net_control users, this leaks internal request IDs, rate-limit headers, and potentially API key prefixes.

**Fix:** Log the full exception; return a generic 502 to the client. **Status: Fixed.**

## Low / Informational

### L2 — Config update accepts arbitrary keys

**File:** `backend/config_mgmt/routes.py:26-35`

`PUT /api/config/{key}` accepts any key name. Admin-only and intentional flexibility, but a confused admin could set typo'd keys (e.g., `scanner_enabled` instead of `scanner.enabled`) with no warning. **Status: Acknowledged, no change.**

### L4 — Audit log has hardcoded 200-entry pagination ceiling

**File:** `backend/audit/routes.py:15`

Items past the most recent 200 are visible only by direct DB query. Audit values (including pre-redaction config changes) persist forever even after they fall out of the visible window. **Status: Acknowledged — the H2/M1 redaction work mitigates the main concern. Pagination beyond 200 is future work.**

### L5 — No `CORSMiddleware` registered

**File:** `backend/app.py`

The app does not register CORS middleware. This is correct for a coupled SPA (same origin in prod, Vite proxy in dev). Worth noting only if external API consumers are ever planned. **Status: Acknowledged, no change.**

### L7 — `python-jose` JWT library has historical CVEs

The app uses `python-jose` for JWTs. Algorithm pinning (`algorithms=[settings.jwt_algorithm]`) prevents the classic `alg: none` confusion. Migration to `PyJWT` would harden against future `jose`-specific issues. **Status: Acknowledged, future work.**

### Roster geojson route — confirmed deleted, no stale references

Verified by `grep -rn "get_session_geojson\|geojson_route\|/geojson"`. No matches outside historical docs. **Status: Verified clean.**

### Manual scan / background scan duplicates

Beyond L1's path mismatch, the two scan code paths duplicate read+import logic. Worth refactoring to share, but not a correctness bug. **Status: Acknowledged, future work.**

### Frontend Sidebar/route role mismatch on Activities

**File:** `frontend/src/layouts/Sidebar.tsx:19`, `frontend/src/App.tsx:72`

Sidebar's Activities link is shown only for `["admin"]`; the route allows `["net_control", "admin"]`. Net_control users can access the page via direct URL but won't see the nav link.

**Fix:** Sync the Sidebar entry to `["net_control", "admin"]`. **Status: Fixed.**

### Frontend Bell polling errors swallowed

**File:** `frontend/src/components/NotificationBell.tsx`

The bell catches all errors during polling and silently ignores them. Intentional (toasting on every poll failure would be noisy), but worth a brief comment. **Status: Acknowledged, comment added.**

### Backend `classify_timing` open-ended branch

**File:** `backend/modules/checkins/service.py:32-34`

Has a `return TimingStatus.EARLY` followed by another `return TimingStatus.EARLY` for the fall-through case. The reviewer flagged this as a bug, but for an open-ended session (no end_date), "before the grace window" is still classifiable as EARLY — there's no LATE state when there's no end. Code is stylistically redundant but behaviorally correct. **Status: Acknowledged. Could merge the branches in a cleanup, no behavioral fix needed.**

### Several backend reviewer "Important" items deemed acceptable

The reviewer flagged sync httpx clients in callbook/delivery backends, missing return-type annotations on route handlers, and helper duplication across `_*_to_response` functions. All are stylistic / minor and not on the critical path. **Status: Acknowledged, deferred.**

## Notes on things that look solid (selected)

- **OAuth state CSRF:** Correctly implemented with `secrets.token_urlsafe(32)` + httponly cookie + `secrets.compare_digest`.
- **PAT tokens:** SHA-256 hashed, plaintext returned only at creation.
- **PAT scope intersection:** Correctly checks both token-granted scopes and current user role rank.
- **No raw SQL:** All queries go through the ORM; the two `sa.text()` calls in `auth/routes.py` use parameterized binds.
- **Role hierarchy enforcement:** `require_role(...)` consistently applied on write endpoints.
- **Path traversal:** `pat_mailbox_path` is admin-configurable only; no user-controlled path construction.
- **No `dangerouslySetInnerHTML`** in any React code.
- **Notification dedupe:** Correctly handles unread-only deduping while letting delivery failures stack.
- **Cookie flags:** `access_token` cookie correctly applies `secure` based on `app_base_url` scheme.
